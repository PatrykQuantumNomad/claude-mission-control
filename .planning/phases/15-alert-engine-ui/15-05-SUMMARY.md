**2. [Rule 1 - Bug] BUG-2: datetime fields serialized without `Z` suffix → JS parses as local time → "in 4 hours" relative-time bug**

- **Issue:** Pydantic v2's default datetime serializer emits naive UTC ISO strings WITHOUT the trailing `Z` suffix (e.g. `2026-05-04T18:45:38` instead of `2026-05-04T18:45:38Z`). Operator at UTC-4 reading a UTC-stamped `fired_at` got "in 4 hours" instead of "just now" / "minutes ago" — broke the ALRT-10 acceptance criterion that the WHEN column accurately reflects fire time.
- **Fix:** Project-wide `UTCDatetime` Annotated type in `cmc/api/schemas/common.py`: `Annotated[datetime, PlainSerializer(_serialize_utc, return_type=str, when_used='json')]`. `_serialize_utc(value)` attaches `UTC` tzinfo when missing and emits ISO-8601 with `Z` suffix. **`when_used='json'` is load-bearing** — gates the serializer so router-level `model_dump()` calls (used to feed SQLAlchemy INSERT/UPDATE) still yield native `datetime` objects; only HTTP JSON responses get the Z suffix. Migrated 8 schema files / 37 datetime fields: `alerts.py` (4), `hitl.py` (3), `mcp.py` (1), `observability.py` (2), `schedules.py` (3), `sessions.py` (6), `skills.py` (2), `tasks.py` (5).
- **Files modified:** `backend/cmc/api/schemas/common.py` (+22 lines, NEW UTCDatetime + _serialize_utc), 8 schema files (datetime → UTCDatetime imports + field migrations), `backend/tests/test_alerts_router.py` (+1 regression test).

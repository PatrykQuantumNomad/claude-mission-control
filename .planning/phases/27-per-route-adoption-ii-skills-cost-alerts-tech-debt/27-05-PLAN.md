**Step 3 — compute prior-period data.** Fetch the WIDER window (the existing `useCostBreakdown('project', range)` already returns up to 30d). For the "prior period" delta:

- If current range is `'7d'`, prior period = the 7 days BEFORE the current 7d window
- If current range is `'30d'`, prior period = the 30 days BEFORE the current 30d window
- If current range is `'1d'` or `'14d'`, mirror the same "same-length-prior" pattern

The data source today is `useCostBreakdown('project', range)`. Two options:

- **(a) Server-side prior fetch:** call `useCostBreakdown` TWICE — once for current, once with an offset shifted backward. Requires backend support for an `offset` param OR a date-range param. Backend extension OUT OF SCOPE per Phase 27 guardrails.
- **(b) Client-side prior fetch (single call):** call `useCostBreakdown('project', '30d')` regardless of current range, slice CLIENT-SIDE into current window + prior window. Same architectural shape as Phase 26 Plan 07's TokenUsageCard overlay (which slices a 30d fetch into current week + prior week client-side).

Recommend **(b)** — matches Phase 26 precedent, no backend change. Pre-fetch 30d, slice client-side per range:

```typescript
import { useMemo } from 'react'
const breakdown30 = useCostBreakdown('project', '30d')
// breakdown30.data is an array per-project rolled up over the LAST 30 days; for client-side prior-period DELTA computation we need a TIME-BUCKETED variant.
// If useCostBreakdown returns only the rolled-up totals (no per-day bucketing), client-side slicing is NOT possible — the prior-period delta can't be computed from the current API.
```

**CRITICAL — if `useCostBreakdown` does NOT return time-bucketed data**, the prior-period overlay is NOT computable client-side without a backend extension. Three escape paths:

- (i) Document the limitation and ship the CompareToggle as a no-op visual demo (toggle changes URL but no DeltaPill renders) — SC#2 partially satisfied; flag as Accepted Exception in 27-VISUAL-CHECK.md
- (ii) Add a tiny backend extension to expose date-bucketed cost-by-project data (RESEARCH guardrail says "no backend SkillRange widening" but does not explicitly bar cost-bucketed expansion; check with operator before doing)
- (iii) Use a sibling endpoint (`useTokens` / `useDailyCost` / equivalent) that DOES have bucketed data; aggregate on the client per-project

**Read `lib/queries.ts` + `lib/api.ts` at execution time** to confirm `useCostBreakdown`'s response shape. If it's already bucketed, ship option (b) verbatim. If not, ship escape (i) and document — the CompareToggle infrastructure lands; the column rendering is gated on data-shape compatibility.

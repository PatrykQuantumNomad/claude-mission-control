## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Filename rename: `skills.$name.tsx` → `skills_.$name.tsx` (TanStack Router parent-layout opt-out)**
- **Found during:** Task 2 (route file creation + first browser smoke test)
- **Issue:** Plan declared `frontend/src/routes/skills.$name.tsx` (the standard nested file-based syntax). When created with that name, navigating to `/skills/analyze` resulted in the detail page never rendering — TanStack treated the file as a CHILD of `skills.tsx`, but `skills.tsx` renders its own page body (no `<Outlet/>` element). Without an Outlet in the parent, the child route can't mount. Either skills.tsx needed an Outlet (would break the existing /skills page that renders multiple panels in a `cmc-card-grid`) or the child file needed the documented trailing-underscore opt-out from parent-layout nesting.
- **Fix:** Renamed the file to `skills_.$name.tsx`. Per TanStack file-based routing docs, the trailing underscore on a path segment opts that file OUT of being treated as a child of the same-stem parent. Public URL is unchanged (still `/skills/$name`); only the filesystem name differs. Re-ran `pnpm build` to regenerate `routeTree.gen.ts`. Browser navigation now resolves the detail page correctly.
- **Files modified:** Renamed `frontend/src/routes/skills.$name.tsx` → `frontend/src/routes/skills_.$name.tsx`; `frontend/src/routeTree.gen.ts` re-regenerated.
- **Verification:** `pnpm build` succeeded; `grep -cF '$name' frontend/src/routeTree.gen.ts` → 13 (≥1 required); browser navigation to `/skills/analyze` rendered all 3 sections; `pnpm tsc --noEmit` clean.
- **Committed in:** `f1006b4` (Task 2 commit)

**2. [Rule 1 - Bug] SkillCostCardForTopSkill wrapper forwarded literal `'(none)'` to per-name endpoint when usage was empty (Plan 14-04 carry-over hotfix)**
- **Found during:** Task 3 human-verify on `/skills` (browser network-tab inspection)
- **Issue:** `SkillCostCardForTopSkill` (Plan 04 D-07 inline wrapper in routes/skills.tsx) had logic equivalent to `<SkillCostCard name={usage.data?.rows?.[0]?.skill_name ?? '(none)'}/>`. When `useSkillUsage('14d', 1)` returned an empty rows array (zero skill_activated events in the production DB at verification time), the wrapper forwarded the sentinel string `'(none)'` to `<SkillCostCard name='(none)'/>`, which fired GET `/api/skills/(none)/cost?range=14d`. The backend correctly 400'd because `'(none)'` doesn't match `_SKILL_NAME_RE`. Network tab showed a red 400, polluting the operator's view of /skills.
- **Fix:** Rewrote the wrapper to read `usageQuery.data?.rows?.length` first; if 0, render an empty `PanelCard` tied to the usageQuery itself (`reqId="SKLP-02"`, `empty.when={() => true}`, empty caption explaining the state) — NEVER forward a sentinel name. Only when a top row exists, forward `name={top.skill_name}` to `<SkillCostCard/>`. Same fix pattern is now documented in `patterns-established` for any future top-N wrapper.
- **Files modified:** `frontend/src/routes/skills.tsx` (+15/-7 lines)
- **Verification:** Browser network tab on /skills no longer shows /api/skills/(none)/cost requests; empty PanelCard renders cleanly; `pnpm tsc --noEmit` clean; `pnpm vitest run` 255/256 (unchanged baseline).
- **Committed in:** `c015c45` (Task 3 hotfix commit, landed under 14-05 scope rather than reopening Plan 14-04)

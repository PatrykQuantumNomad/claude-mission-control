## Decisions Made

- **`isolation_level = None` instead of `.autocommit = True`** — RESEARCH.md Pattern 1 and Pitfall 1 documented the toggle as `dbapi_connection.autocommit = True`, which works against the synchronous sqlite3 DBAPI. Under `sqlite+aiosqlite`, the dbapi connection inside the connect listener is SQLAlchemy's `AsyncAdapt_aiosqlite_connection` wrapper — that adapter exposes `.isolation_level` (not `.autocommit`). Setting `.isolation_level = None` on the underlying sqlite3 driver is the documented autocommit-equivalent. Verified by direct test: `PRAGMA foreign_keys` returns `1` after the toggle.
- **Add `greenlet>=3.0` to dependencies** — Plan 02 locked the SQLAlchemy stack but omitted greenlet. SQLAlchemy 2.0 async IO uses greenlet to bridge sync DBAPI calls into the async event loop, and raises `ValueError("the greenlet library is required to use this function. No module named 'greenlet'")` on the first `engine.connect()` call without it. This is a Plan 02 blind-spot, surfaced here as a Rule 3 (blocking issue) fix.
- **`migrations/README` retained** — `alembic init -t async` generates this one-liner. Not in the plan's `files_modified`, but tracked anyway because untracked scaffold output creates persistent gitnoise; the file is harmless and identifies the directory's purpose.
- **`alembic.ini` `sqlalchemy.url = sqlite+aiosqlite:///data/cmc.db` (relative)** — The CLI fallback URL is intentionally relative; Plan 06's lifespan will inject the repo-root-anchored absolute URL via `config.attributes['connection']`, sidestepping cwd drift. For developers running `alembic upgrade head` directly from `backend/`, the relative path resolves to `backend/data/cmc.db`, which is consistent with `cmc.config.Settings.db_path` defaulting to `<repo_root>/data/cmc.db` (since `repo_root() == <project>` and `backend/data/cmc.db` doesn't match — note this is documented as a deferred-issue below; Plan 06 is the canonical entry point).

**2. [Rule 1 - Bug] `dbapi_connection.autocommit` does not exist on the aiosqlite adapter**

- **Found during:** Task 1 GREEN phase (running new tests after implementing the engine module).
- **Issue:** RESEARCH.md Pattern 1 / Pitfall 1 wrote the connect listener as `prior_autocommit = dbapi_connection.autocommit` followed by `dbapi_connection.autocommit = True`. That works for the synchronous sqlite3 DBAPI, but under `sqlite+aiosqlite` SQLAlchemy wraps the connection in `AsyncAdapt_aiosqlite_connection`, which does NOT expose `.autocommit`. The first connect raised `AttributeError: 'AsyncAdapt_aiosqlite_connection' object has no attribute 'autocommit'`.
- **Fix:** Translated the toggle to the equivalent `isolation_level` semantics. Setting `dbapi_connection.isolation_level = None` puts the underlying sqlite3 driver into autocommit mode (sqlite3's documented behavior). The toggle is wrapped in try/finally so the prior `isolation_level` is always restored. Verified by isolated reproduction — both `PRAGMA journal_mode` (returns `wal`) and `PRAGMA foreign_keys` (returns `1`) succeed against this approach.
- **Files modified:** `backend/cmc/db/engine.py`. Updated the module docstring to explain the translation.
- **Verification:** `pytest -k "wal or foreign_keys"` — 2/2 pass. The `test_foreign_keys_enabled` test is the regression guard for Pitfall 1; it is now green.
- **Committed in:** `d3c9e90` (Task 1 GREEN commit, with the fix included from the start).

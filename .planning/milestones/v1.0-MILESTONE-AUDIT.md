## Tech Debt (no blockers)

### Documented v1 Design Decisions
- **No frontend `useRejectTask`** — Phase 10 scoped reject to Telegram only. Dashboard "delete" sets status=removed; Telegram "reject" sets status=cancelled. Different terminal states by design; cross-channel parity deferred.
- **`FollowUpPump` queue keyed on task_id** while dashboard live-message endpoint writes session_id-keyed queue. v1 dispatcher tasks don't receive dashboard-originated follow-ups; documented in `run_stream.py:254` and `follow_ups.py:14-15`.
- **`reply_inbox` Telegram callback** is a NOOP (handler acks "reply via dashboard"). Documented v1 limitation.
- **No CI gate for `npm run test:e2e`** — Q6 LOCKED in Phase 9; manual gate by design.

### Environmental / Cosmetic
- `test_phase3_system.py::test_sapi05_firehose_route_is_registered` fails only on Python 3.11; project pins ≥3.13.
- `test_phase4_estop` pre-existing flake (logged in deferred-items.md).
- Orphan api.ts client functions: `mcpSync`, `mcpMeasure`, `sync`, `createDecision`, `createInbox`, `sessionLiveState` — exported but unused. Cosmetic.

### Pending Architectural Refactor (post-v1)
- `answer_poll.py` uses 2s DB polling; a file-watch implementation would be required if notifier/dispatcher ever splits remote.
